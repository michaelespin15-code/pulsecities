## 3. Data threads (findings only)

**177 vs 178 ZIPs — not a dropped real ZIP.** `neighborhoods` holds 178 distinct ZIPs; the only one absent from `displacement_scores` is sentinel `99999` (`name = NULL`, zero rows in `permits_raw` / `complaints_raw` / `evictions_raw`). Scoring's valid-ZIP universe (see commit `0f0aed6`) correctly excludes it, so **177 scored is correct** — no join/geometry/filter defect. No action.

**dob_permits low counts — explained, query is correct.** Run `id 821` (2026-06-09) processed 3; the 2026-06-10 run processed 2. The scraper filters by `filing_date LIKE '%/YYYY'` for the last 3 years plus `permit_type`, **not** by watermark (`scrapers/permits.py:160-165`); the watermark is informational only and dedup is via the unique constraint with `on_conflict_do_nothing`. So `records_processed` counts only genuinely new inserts. `MAX(filing_date)` in `permits_raw` = 2026-06-06 = the stored watermark, i.e. the source has published nothing dated after 6/06. The 2-3 inserts are the few late-arriving alteration permits. Low counts reflect upstream filing lag, not a wrong query.

**Duplicate runs within 60s — only dof_assessments, all failures.** Every night `dof_assessments` writes 2-3 `scraper_runs` rows ~30-60s apart, all `status=failure`. These are tenacity retry attempts (`pipeline._run_scraper_with_retry`, `stop_after_attempt(3)`), each opening a fresh session and writing its own run row. No other scraper shows duplicates. This is the known/deferred `dof_assessments` failure; side effect is up to 3 failure rows/night inflating `scraper_runs`. No fix requested.
