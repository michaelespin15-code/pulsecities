# MTEK Soft 404 Diagnosis

Read-only investigation on 2026-06-23 over the gunicorn socket. No code changed.

## Summary

`/operator/mtek-nyc` is flagged Soft 404 while `/operator/phantom-capital` and `/operator/bredif` are not. The investigation shows there is **no MTEK-unique defect in the served markup or a broken render**: the served HTML body is byte-identical across all three operators (and across a non-existent operator), and the API returns full data for all three. The soft 404 is a property of the rendering architecture, not of MTEK's content. Every operator page ships a client-rendered shell that returns HTTP 200 with a "Loading" placeholder and a hidden "Operator not found" block, and injects all real content via JS after an API call. MTEK is the URL Google currently classifies, and the one concrete current difference that singles it out is its **stale SSR meta** (advertises 26 acquisitions / 7 LLCs while the page renders 37 / 9), the only operator whose head metadata contradicts its own body.

## 1. Raw served HTML diff (JS bypassed, over the socket)

All three return HTTP 200, ~70.4 KB, and the **bodies after `</head>` are byte-identical**:

```
mtek vs phantom body identical?  YES
mtek vs bredif  body identical?  YES
phantom vs bredif body identical? YES
```

The only differences are in the `<head>` meta: operator name and counts.

| Operator | SSR meta (title/description) | Live API (rendered body) | Meta matches body? |
|---|---|---|---|
| mtek-nyc | "MTEK: 26 property acquisitions ... 7 LLC entities" | 37 acquisitions, 9 LLCs | **NO** |
| phantom-capital | "PHANTOM: 65 ... 32 LLC entities" | 65 acquisitions, 32 LLCs | yes |
| bredif | "BREDIF: 67 ... 3 LLC entities" | 67 acquisitions, 3 LLCs | yes |

MTEK is the only operator whose served meta disagrees with what the page renders. Source of the stale numbers: `scripts/operator_network_analysis.json` (cluster JSON, MTEK = 26 acq / 7 LLC) feeds the SSR meta, while the operators table and the API return 37 / 9.

## 2. SSR path (api/routes/frontend.py, operator_page)

The handler does **not** server-side render the operator's data into the body. It serves the static `frontend/operator.html` shell and injects only `<head>` tags (title, description, canonical, og/twitter). The body is the same shell for every operator. The shell's initial state, present identically in all three served pages:

- `#loading-state { display: flex; }` with visible text "Loading operator profile..."
- `#content { display: none; }` (all real content hidden until JS runs)
- `#error-state { display: none; }` containing the strings "Operator not found" and "This operator profile is not available yet"
- empty `<h1 id="op-root"></h1>` and empty `<div class="stat-val" id="stat-*"></div>` placeholders

So a crawler that indexes the pre-render HTML, or whose JS render does not complete, sees: a "Loading" spinner, an empty headline, empty stats, and embedded not-found text. That reads as soft 404.

Slug resolution is fine and identical for all three: `mtek-nyc` resolves to operator_root MTEK, `phantom-capital` to PHANTOM, `bredif` to BREDIF; each gets its meta injected. MTEK does **not** fall through to a not-found render server-side.

## 3. The architecture returns 200 for everything, including non-existent operators

```
GET /operator/this-operator-does-not-exist-xyz  ->  HTTP 200
   body identical to the real /operator/mtek-nyc page (contains "Operator not found", "Loading operator profile")
```

A bogus operator and a real operator serve the identical 200 shell. The not-found vs real decision happens only in client JS; the HTTP status never reflects 404. This is the textbook soft-404 generator: Google must execute JS to tell a real operator page from a missing one, and the pre-render HTML always looks like a loading/not-found placeholder.

## 4. API is healthy (confirmed)

All three return 200 with full, structurally identical payloads:

```
mtek-nyc:        total_properties=37 total_acquisitions=37 llc=9  recent_acquisitions=39 eviction_then_buy=1
phantom-capital: total_properties=64 total_acquisitions=65 llc=32 recent_acquisitions=50 eviction_then_buy=4
bredif:          total_properties=66 total_acquisitions=67 llc=3  recent_acquisitions=50 eviction_then_buy=6
```

No nulls, no missing keys for MTEK. The client render path produces a full page for MTEK exactly as for the other two. So the soft 404 is not a render break on MTEK's data.

## Why MTEK and not BREDIF/PHANTOM

There is no MTEK-only element in the served body and no broken render, so nothing in the current markup deterministically forces MTEK to fail where the others pass. What distinguishes mtek-nyc:

1. **Inconsistent, stale head metadata (the one concrete current difference).** MTEK is the only operator whose SSR title and description (26 acquisitions, 7 LLCs) contradict the rendered body (37, 9). Stale, self-contradicting metadata is a negative quality signal and weakens Google's confidence that the URL is a substantive, consistent resource. PHANTOM and BREDIF metadata happen to match their live data, so they do not send that mixed signal.
2. **Weakest stored signals.** Every operator page is a soft-404 candidate by architecture. mtek-nyc is the URL Google has currently tipped over the line. The page that is borderline for all three only needs a small negative nudge on one of them, and MTEK is the one carrying it (point 1), so it is the first to be classified while the other two sit just on the passing side of the same heuristic.

The practical reading: this is a rendering-architecture problem that exposes a hidden not-found shell to crawlers on every operator URL. MTEK tripped first. PHANTOM and BREDIF are latent risks under the same design.

## Exact element/string that reads as not-found to a crawler

In the served body of every operator page (200 status), the only visible content before JS is:

```html
<div id="loading-state" ... >Loading operator profile...</div>
```

with `#content { display:none }` and a hidden `#error-state` block containing:

```html
<div ...>Operator not found</div>
<div ... id="error-msg">This operator profile is not available yet.</div>
```

No operator name, stats, or acquisitions exist in the body until client JS replaces this shell.

## Recommended fix (do not implement yet)

Make the served HTML reflect the true state without depending on client JS:

1. **SSR-inject the operator's real data into the body** in `api/routes/frontend.py` `operator_page()`: the H1 operator name, the stat numbers (properties, acquisitions, LLCs, ZIPs), and at least the recent-acquisitions rows. Keep the existing JS as progressive enhancement. This gives crawlers substantive content on first byte.
2. **Stop emitting the not-found/empty shell on 200 pages.** Return HTTP 404 (and a real not-found body) when the slug does not resolve to a class 'operator' row, instead of 200 + the hidden error block. Today every `/operator/*` returns 200, including non-existent slugs, which is itself a soft-404 source.
3. **Fix the stale meta** by sourcing the title/description counts from the operators DB row (`total_acquisitions`, `jsonb_array_length(llc_entities)`) rather than `scripts/operator_network_analysis.json`. This makes MTEK read 37 / 9 and aligns head metadata with rendered content for every operator.

Items 1 and 2 remove the soft-404 ambiguity for all operator pages; item 3 removes the specific inconsistent signal that singled out MTEK.
