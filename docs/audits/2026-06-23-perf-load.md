# PulseCities Performance and Load Audit

Read-only audit on 2026-06-23. Box: 3.8 GB RAM, 2 vCPU, 2 gunicorn (uvicorn) workers. Tools: curl over the gunicorn unix socket, ApacheBench (`ab`) against the public host. No app, schema, or data changes. Service restarted once at the start to clear per-worker caches for honest cold timings.

## Headline

The box is **CPU/worker-bound, not memory-bound** (memory stayed flat at ~890 MB used / 2.6 GB free under load; load average rose to 3.89 on 2 cores). Cached and static endpoints sustain 180-340 req/s from a single client. Three things will hurt under a press spike, in order: (1) **there is no CDN in front** (Cloudflare is DNS-only, every request hits origin), (2) **the operator detail page never hits its cache** on the canonical slug URL and recomputes DB queries on every request (74 req/s ceiling), and (3) **top-risk recomputes a heavy multi-table aggregation on cache miss** (2.8-20s) that blocks a worker. The per-IP rate limit that would cap a flood **is not actually enforcing**.

## Section 2 - Concurrency / load (~50 concurrent, 20s each, public host through nginx)

| Endpoint | req/s | p50 | p95 | p99 | non-2xx |
|---|---|---|---|---|---|
| `/` (nginx static) | 337 | 141 ms | 221 ms | 264 ms | 0 |
| `/map` (nginx static) | 299 | 158 ms | 243 ms | 288 ms | 0 |
| `/operator/mtek-nyc` (SSR via gunicorn) | **74** | **651 ms** | **960 ms** | **1097 ms** | 0 |
| `/api/stats` (cached) | 181 | 260 ms | 445 ms | 587 ms | 2 |
| `/api/neighborhoods` (cached) | 192 | 241 ms | 471 ms | 731 ms | 2 |
| `/api/neighborhoods/top-risk` (cached) | 194 | 251 ms | 415 ms | 507 ms | 2 |

- Static pages scale (nginx serves them without touching gunicorn).
- Cached API endpoints sustain ~190 req/s from one client; the work is JSON serialization + proxy, no DB.
- **`/operator/mtek-nyc` is the outlier at 74 req/s with ~1s p99.** It is served by gunicorn and, because of a cache-key bug (Section 3), recomputes the full SSR plus two DB queries on every request. Per-request worker cost ~27 ms, and at 50 concurrency against 2 workers latency balloons to ~650 ms (queueing).
- The `2` non-2xx on each API endpoint are isolated blips, not a rate-limit wall (the limiter does not enforce, see Section 3).

### Box behavior under load
- **Memory: flat.** ~841 MB used at idle, ~890 MB peak, 2.6 GB always available, swap untouched. Not memory-bound.
- **CPU: saturated.** Load average 0.43 idle to 3.89 peak on 2 cores. CPU/worker throughput is the ceiling. (Caveat: `ab` runs on the same box and inflates load; real external load would push slightly less, but the origin is still CPU-bound on gunicorn work.)
- **Workers: stable.** No restarts, no `WORKER TIMEOUT`, no SIGTERM during the run. `--workers 2`, no `--max-requests` recycling.

## Section 3 - Root causes

1. **No edge cache / no CDN in front (biggest spike risk).** `pulsecities.com` resolves to `104.236.87.19`, the droplet's own IP; responses carry `server: nginx` with no `cf-ray` / `cf-cache-status`. Cloudflare is DNS-only. Every request, static and API, hits the 2-worker origin. The API already sends `cache-control: public, max-age=3600` (stats) and `max-age=82800` (neighborhoods/top-risk), so the data is edge-cacheable, but nothing is caching it. A press spike is many unique IPs, all landing on origin.

2. **Operator page cache key mismatch (`api/routes/frontend.py`).** The cache is read with the raw path token and written with the canonical root:
   - line 629: `cached = _op_page_cache.get(root_upper)` where `root_upper = root.upper().strip()` -> `"MTEK-NYC"` for the slug URL.
   - line ~656: `root_upper = op_row.operator_root` reassigns it to `"MTEK"`.
   - line 792: `_op_page_cache[root_upper] = (...)` writes under `"MTEK"`.
   So `/operator/mtek-nyc` (the canonical, sitemap URL) reads key `MTEK-NYC`, writes key `MTEK`, and **never hits**. Every request recomputes the SSR body, including the two DB queries added for SSR content (distinct ZIP count + 20 recent acquisitions). The root-name URL `/operator/MTEK` does hit (same key both ways), but crawlers and links use the slug. This is the 74 req/s result.

3. **top-risk cold recompute (`api/routes/neighborhoods.py:227`).** On cache miss it runs a multi-CTE live aggregation over `ownership_raw` joined `parcels` (plus permits/evictions/complaints/violations) with several `NOT ILIKE '%MORTGAGE%'`-style negative filters and a `NOT EXISTS` subquery. Negative `ILIKE` and `NOT EXISTS` cannot use the existing indexes (including the `party_name` trigram index), so this is largely sequential scans over 154k+ ownership rows. Cost is 2.8s warm-buffers, up to 9-20s cold-buffers. It is cached per worker for 23h (`_TOP_RISK_TTL = 82800`), so it normally only runs at boot, but **on TTL expiry the worker that catches it blocks for the full query** while other requests queue on the remaining worker (a mini cache stampede). The data it derives is already computed nightly into `displacement_scores`; re-deriving raw counts live is redundant.

4. **Per-worker in-memory caches, no shared store.** `_op_page_cache`, `_TOP_RISK_CACHE`, `_GEOJSON_CACHE`, `_STATS_CACHE` are module-level dicts. With 2 workers, every cache is warmed twice (2x DB recompute at boot) and held twice (2x memory), and each worker recomputes independently on its own TTL expiry. There is no stale-while-revalidate, so a miss is a synchronous recompute on the request path.

5. **Rate limiter is not enforcing.** 80 sequential requests to `/api/stats` returned 80x 200; the 50-concurrent runs produced no 429 wall. The `x-ratelimit-*` headers are emitted (`headers_enabled=True`) but the limit does not bind. Cause: `Limiter(key_func=get_remote_address)` keys on the client IP, but nginx proxies to gunicorn over a **unix socket**, so `request.client.host` has no real IP, and `ProxyHeadersMiddleware(trusted_hosts=["127.0.0.1"])` only trusts a 127.0.0.1 peer, not a socket peer, so `X-Forwarded-For` is not applied. The key resolves to an empty/constant value and the limit effectively no-ops. Each route module also constructs its own `Limiter` instance separate from `app.state.limiter`. Net effect: no per-IP flood protection (and the load numbers above are true capacity, not rate-limited).

6. **stats cold ~2s (`api/routes/stats.py`).** Cache miss runs `COUNT(*)` across the large source tables plus `MAX(date)` freshness probes. Warmed at boot, 1h TTL. Lower priority because it is small and pre-warmed, but it is a synchronous recompute on miss like top-risk.

## Prioritized hardening list (implement nothing here)

- **P0 - Fix the operator-page cache key.** In `operator_page()`, read and write `_op_page_cache` under the same canonical key (resolve the slug/root to the canonical id first, then use it for both `get` and the final `set`). Atomic, code-only. Restores `/operator/{slug}` from 74 req/s to static-like throughput and removes the per-request DB queries. This is the page a press story most directly drives traffic to.
- **P0 - Put Cloudflare in front (orange-cloud) and add a Cache Rule for the cacheable GETs.** Proxy the domain and add an edge Cache Rule (or "Cache Everything") for `/api/stats`, `/api/neighborhoods`, `/api/neighborhoods/top-risk` (they already send `public, max-age`), plus the static pages. Single biggest lever: converts a multi-IP spike into edge hits and shields the 2-worker origin. Infra change (Michael).
- **P1 - Make top-risk non-blocking on miss.** Either serve stale-while-revalidate (return the existing cached payload immediately and refresh in a background task) or read the precomputed values from `displacement_scores` instead of re-deriving raw counts live. Removes the 3-20s worker stall on TTL expiry.
- **P1 - Add a Postgres `statement_timeout` on the API DB session** (e.g. 5s). Bounds any pathological query so one slow recompute cannot pin a worker for 20s during a spike.
- **P1 - Decide and fix rate limiting.** Either set gunicorn `--forwarded-allow-ips` / configure `ProxyHeadersMiddleware` so `get_remote_address` resolves the real client IP (so 60/min actually binds), or rely on Cloudflare rate limiting once it is in front. Today there is no working per-IP cap.
- **P1 - Bump gunicorn workers 2 -> 4.** RAM is ample (2.6 GB free). More workers absorb blocking DB waits and raise concurrency for the uncached/SSR paths. Pair with a shared cache (next item) so the extra workers do not multiply cold recomputes and memory.
- **P2 - Move hot caches to a shared store (Redis).** One warm copy of top-risk/stats/neighborhoods across all workers: one recompute instead of N, less memory, and no per-worker stampede.
- **P2 - Pre-warm the flagship operator pages at boot** (MTEK, PHANTOM, BREDIF) once the cache-key fix lands, so the first press visitor is not the one paying the SSR + DB cost.

## Method notes
- Latency over the unix socket isolates app/DB cost from nginx/TLS.
- Load test hit the public host (real nginx path). Cloudflare being DNS-only means "public" already equals "origin" here, so these numbers are origin capacity. Volume kept modest (50 concurrent, 20s/endpoint) to avoid self-DoS; `ab` ran on-box, which inflates the measured load average.
- All Section-2 endpoints are nginx-static or in-app-cached, so the load test did not hammer the database; the DB-bound risk (operator detail, score, search, pulse) is characterized from code and single-request timings, not flooded.
